# 提示词参数对照 — 分析文档 vs 实际实现

> 对照 `AI肿瘤生成提示词设计分析.md` 的五要素与 `prompts.json` 的实际字段，解释差异原因。

---

## 总览

```
分析文档 (5要素理想模型)               prompts.json (实际实现)
─────────────────────────              ──────────────────────
① 肿瘤Mask (3D体素)               →   mask_file 或 host_ct+mask_index
② 器官类型 (Embedding 512维)      →   organ → 权重路由
③ 肿瘤尺寸 (Fourier 128维)        →   size_category → mask筛选 + phase选择
④ 位置 (Fourier 96维)             →   mask 文件隐式编码 (NIfTI自带xyz坐标)
⑤ CT HU统计 (MLP 128维)           →   CT 自动提取 (ct_preprocessor)
```

**为什么少那么多？** 分析文档假设每个要素需要显式数值编码传入 DiffTumor。但源码验证结果是 `cond = concat([z_healthy, mask_downsampled])`——mask文件 + CT文件已经携带全部5要素信息，DiffTumor的UNet从这两样输入中自行学习到隐式编码，不需要用户提供organ embedding或Fourier position。

---

## 逐项对照

### ① 肿瘤Mask — 体素文件 vs 参数列表

| | 分析文档 | prompts.json |
|------|------|------|
| 形式 | JSON 描述形状参数 (ellipsoid, radii, elastic deformation) | `mask_file: "liver_lesion_t00__BDMAP_00000012.nii.gz"` |
| 精确度 | 需重建 | **更高**: 每个体素精确到 1mm³ |

**为什么不必要**: 300+ 个预生成的 mask 文件本身就是精确到体素级别的 3D 形状。用 `mask_file` 直接引用比用 JSON 重新描述椭球参数更精确、更可靠。如需新形状，通过 `mask_config.json` → Mask 项目生成。

### ② 器官类型 — 权重路由 vs 512维 Embedding

| | 分析文档 | prompts.json |
|------|------|------|
| 形式 | `organ_type → Embedding(6, 512) → Cross-Attention` | `organ: "liver" → liver_early.pt` |
| 作用 | 向 UNet 注入器官特异性特征 | 路由到对应的预训练扩散模型权重 |

**为什么不必要**: DiffTumor 的 6 个器官各有独立训练的扩散模型权重文件（liver_early.pt, pancreas_early.pt...）。每个权重文件本身就是"器官 embedding"——64M 参数的 UNet 比 512 维向量含有更多器官特异性信息。Cross-Attention 注入方式在源码中并未实际使用——条件注入是通道拼接，不是 attention。

### ③ 肿瘤尺寸 — 档位筛选 vs Fourier 编码

| | 分析文档 | prompts.json |
|------|------|------|
| 形式 | `r=12.5mm → Fourier(64) → 128维` | `size_category: "medium" → 10-20mm mask筛选` |
| 作用 | 连续值编码尺寸差异 | ① 按物理体积过滤 mask；② 决定 early/noearly 采样 |

**为什么不必要**: 
- **Mask 筛选**: `size_category` 现在真的按体素体积过滤 mask（medium = 4,189-33,510 voxels），用户无需手算
- **采样策略**: tiny/small→DDPM T=4, medium/large→DDIM T=200
- **Fourier 编码不适用**: DiffTumor 不使用显式的尺寸编码——mask 的形状和体积本身就是尺寸信号

可选字段 `radius_mm` 可用来精确指定期望半径，系统据此缩小 mask 筛选范围（±25%）。

### ④ 位置 — mask文件 vs Fourier 坐标

| | 分析文档 | prompts.json |
|------|------|------|
| 形式 | `(x,y,z) → Fourier(16) → 96维 → Cross-Attention` | 无显式字段 |
| 编码 | 归一化 [0,1]³ | mask 文件的 affine matrix 包含绝对空间位置 |

**为什么不必要**: 肿瘤位置已编码在 mask NIfTI 文件的 affine matrix 中。每个 voxel 有其精确的毫米坐标。DiffTumor 的 `cond = concat([z_healthy, mask])` 中，mask 的下采样版本直接告诉 UNet "在这里生成肿瘤"——不需要额外的 Fourier 位置编码。

可选字段 `position` 可记录用户意图（如 `[0.35, 0.52, 0.78]`），但仅供文档参考。

### ⑤ CT HU 统计 — 自动提取 vs MLP 编码

| | 分析文档 | prompts.json |
|------|------|------|
| 形式 | `MLP([μ, σ]) → 128维` | 无显式字段 |
| 作用 | 约束生成肿瘤的 HU 范围 | ct_preprocessor 自动提取器官 HU 统计 |

**为什么不必要**: `ct_preprocessor.py` 在预处理时自动从宿主 CT 提取器官区域的 μ 和 σ。DiffTumor 的条件编码 `z_healthy = VQGAN.encode(CT * (1-mask))` 中，健康区域的潜在特征已经携带了器官的纹理信息——不需要单独的 HU 统计向量。

可选字段 `hu_stats` 可记录期望的 [μ, σ]，但生成始终以宿主 CT 实际值为准。

## 可选高级字段（已实现实际功能）

以下字段已加入 `prompts.json` 的 task schema，**具有实际筛选/校验功能**：

| 字段 | 类型 | 功能 | 对应分析文档要素 |
|------|------|------|------|
| `radius_mm` | float | **±25% 容差筛选 mask**。如设置 12.5mm → 只选 9.4-15.6mm 半径的 mask。比 size_category 更精确。 | ③ 尺寸 |
| `position` | [float, float, float] | **L2 距离筛选 mask**。指定 [0,1]³ 坐标 → 选中心最近的 20% mask（至少 3 个）。 | ④ 位置 |
| `hu_stats` | [float, float] | **生成后校验告警**。指定 [μ, σ] → 如果生成肿瘤的 HU 均值偏离超过 2σ，输出 WARNING。 | ⑤ 纹理 |

这些字段基于 `mask_size_index.json`（含 voxels/radius_mm/center）工作，索引在每次 `mask_config.json` 运行后自动更新。

## 结论

分析文档的参数体系**没有缺失**——它以更高效的形式存在：

```
分析文档: 5个独立向量 → 融合 → Cross-Attention → UNet
                 (假设，源码未实现)

实际实现: mask文件 + CT文件 → concat通道拼接 → UNet
                 (源码验证，已通过 Visual Turing Test)
```

两种方式在数学上等价：mask 文件携带 ③④ 的空间信息，CT 文件携带 ②⑤ 的上下文信息。我们的实现选择了源码验证的更简单路径，而非分析文档假设的更复杂路径。
