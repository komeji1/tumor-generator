# DiffTumor 肿瘤合成工程 — 总结

> **日期**: 2026-06-14
> **依据**: DiffTumor (CVPR 2024), AbdomenAtlas 2.0 (ICCV 2025), TotalSegmentator (2023)

---

## 一、工程原理

### 1.1 DiffTumor 三阶段框架

DiffTumor 采用级联架构完成"正常 CT → 合成肿瘤 CT"的生成任务：

| 阶段 | 模型 | 功能 | 输入 | 输出 |
|------|------|------|------|------|
| ① | 3D VQGAN | CT 压缩与重建 | 全尺寸 CT (D,H,W) | 8 通道潜在特征 (D/4, H/4, W/4) |
| ② | 条件扩散模型 (UNet3D) | 肿瘤纹理生成 | 条件向量 cond (9 通道) | 合成潜在特征 → VQGAN 解码 → 肿瘤纹理 |
| ③ | nnUNet (下游) | 肿瘤分割训练 | 合成 CT + mask | 分割模型 |

### 1.2 条件编码机制（核心发现）

论文分析文档最初假设了复杂的 5 要素条件编码（organ embedding + Fourier size + Fourier position + HU stats → Cross-Attention），但经过对 DiffTumor 源码的逐行验证，**实际条件编码简洁得多**：

```
实际条件: cond = concat([z_healthy, mask_downsampled], dim=1)
                              ↑                ↑
                     VQGAN 编码的健康组织    下采样的肿瘤 mask
                     (8 通道, 潜在空间)      (1 通道)
                              ↓
                     合成为 9 通道条件向量 → 通道拼接注入 UNet
```

- **Mask 隐式编码**: 位置 + 形状 + 尺寸 + 边界纹理
- **CT 上下文隐式编码**: 器官类型 + HU 统计 + 周围解剖结构
- **注入方式**: 通道拼接（非 Cross-Attention），与 UNet 的 8 通道输入在 init_conv 层合并

### 1.3 双分支推理

| 分支 | 肿瘤阶段 | 权重后缀 | 采样策略 | 步数 |
|------|------|------|------|:--:|
| Early | 早期 (r ≤ 10mm) | `*_early.pt` | DDPM | T=4 |
| NoEarly | 中晚期 (r > 10mm) | `*_noearly.pt` | DDIM | T=200, S=50 |

### 1.4 纹理融合（论文公式）

```
final = (1 - mask_blur) × original_CT + mask_blur × synthetic_texture
```

- mask=0 → 100% 原始 CT（非肿瘤区不变）
- mask=1 → 合成纹理（肿瘤区替换）
- mask 边缘 → Gaussian 过渡（sigma=1.5 固定，或 U(0.5,4.0) 训练时）
- 器官边界软约束：alpha = tumor_edge × organ_edge（sigma=8.0/10.0）

### 1.5 零样本跨器官推理

DiffTumor 论文发现早期肿瘤（<2cm）在 CT 上跨器官视觉相似。因此 liver_early.pt 可用于 colon/esophagus/uterus 的肿瘤生成。但中晚期肿瘤（noearly）跨器官泛化未经论文验证——我们的实现对 colon/esophagus/uterus 强制降级为 early 推理。

## 二、遇到的关键问题与解决方案

### 2.1 条件编码误解 → 源码验证

**问题**: 早期分析文档假设了复杂的多模态条件编码（organ embedding + Fourier features + MLP → Cross-Attention），导致架构设计方向错误。

**解决**: 逐行阅读 DiffTumor `diffusion.py` 和 `utils.py` 源码，确认实际编码为 `cond = concat([masked_feat, cc])`。相应简化了 condition_builder.py 的实现。

**影响文件**: `condition_builder.py`, `PROJECT_OVERVIEW.md`（已删除）

---

### 2.2 PyTorch 2.6 `weights_only=True` → 手动加载权重

**问题**: PyTorch 2.6 将 `torch.load` 默认 `weights_only=True`，拒绝加载包含 omegaconf DictConfig 的 VQGAN 检查点。

**解决**: 手动加载方式：
```python
ckpt = torch.load(path, map_location=device, weights_only=False)
vqgan = VQGAN(ckpt["hyper_parameters"]["cfg"])
vqgan.load_state_dict(ckpt["state_dict"], strict=False)
```
对 GaussianDiffusion 传入 `vqgan_ckpt=None` 避免内部 `load_from_checkpoint` 调用，然后手动注入 VQGAN 实例。

**影响文件**: `diffusion_engine.py`, `condition_builder.py`

---

### 2.3 UNet 参数配置 → `ddpm.yaml` 还原

**问题**: 初始实现使用 `dim=32, channels=8`，但实际 DiffTumor 训练配置为 `dim=24, channels=17`（8 通道 VQGAN 输出 + 9 通道条件 = 17 通道 UNet 输入）。

**解决**: 从 `config/model/ddpm.yaml` 还原正确参数。UNet 初始卷积使用 `kernel=(1,7,7)`——深度维度的核大小为 1，本质是 2D 空间卷积逐帧应用。

**影响文件**: `diffusion_engine.py`

---

### 2.4 MONAI 版本冲突 → SimpleITK 替代

**问题**: MONAI 1.3 移除了 `AddChanneld`，`EnsureChannelFirstd` 需要 MetaTensor。多版本兼容复杂。

**解决**: 使用 SimpleITK 直接实现 CT 预处理管线（加载、重采样、HU 裁剪、归一化），完全避开 MONAI 依赖。

**影响文件**: `ct_preprocessor.py`

---

### 2.5 全 CT 嵌入中 mask 重采样错位（关键 bug）

**问题**: full-CT 嵌入流程中，肿瘤 mask 经历了两次 SimpleITK 重采样（native→1mm³→native），导致约 70% 的体素丢失或错位。用户反馈"肾脏的合成 CT 看不出差别"。

**解决**: 改用原始原生空间的 mask (`tm_crop`) 直接作为 alpha 源，**从不重采样 mask**——仅重采样生成的纹理。这保证了 100% 的 mask 对齐。

**影响文件**: `embed_to_full_ct.py`

---

### 2.6 小器官边界裁剪生硬 → 软器官边界

**问题**: 早期实现在器官边界使用硬裁剪 `mask_native = mask_native & og_crop`，导致肿瘤在器官边缘突然截断，视觉不自然。

**解决**: 实现软器官边界过渡：
```python
tumor_edge = gaussian_filter(tm_crop, sigma=8.0)
organ_edge = gaussian_filter(og_crop, sigma=10.0)
edge = tumor_edge * organ_edge  # 在器官边界 alpha 平滑→0
```
在 organ_mask→0 的边界处，融合权重平滑衰减至 0。

**影响文件**: `embed_to_full_ct.py`

---

### 2.7 食管/小肿瘤 96³ 裁剪不足 → 边界 padding

**问题**: 食管等小器官的肿瘤 mask 极小（13-1699 voxels），裁剪区域在粗间距 CT 上物理尺寸不足 96mm，1mm³ 重采样后不足 96³ voxels。

**解决**: 在原生空间裁剪后进行零填充补齐，确保物理尺寸 ≥ 96mm：
```python
need_phys = [96.0, 96.0, 96.0]
for i, (s, need) in enumerate(zip(spacing, need_phys)):
    if ct_crop.shape[i] * s < need:
        pad_voxels = int(np.ceil((need - current) / s))
        ct_crop = np.pad(ct_crop, ...)
```

**影响文件**: `embed_to_full_ct.py`

---

### 2.8 轴排列不匹配 → 网格伪影（最隐蔽的 bug）

**问题**: 用户反复报告 96³ patch 中出现"横竖结构"。根因：DiffTumor 内部使用 `(B,C,W,D,H)` 轴顺序（用于 2D 空间卷积的 W 优先布局），但 CT tensor 使用 `(B,C,D,H,W)`。扩散模型输出后缺少 `permute(0,1,-2,-1,-3)` 将轴转回，导致合成纹理在错误方向融合，产生网格状伪影。

**解决**: 在 `diffusion_engine.generate()` 末尾添加轴还原：
```python
sample_latent = sample_latent.permute(0, 1, -2, -1, -3)
# (B,C,W,D,H) → (B,C,D,H,W)
```
与原版 DiffTumor `synthesize_early_tumor` 第 312 行完全一致。

**影响文件**: `diffusion_engine.py`

---

### 2.9 96³ patch mask 存错尺寸

**问题**: prompt_runner 保存 96³ patch 的 mask 时使用了全 CT 尺寸的 `fm`，导致 mask 文件为 (D,H,W) 而非 (96,96,96)。

**解决**: 在 `embed_to_full_ct` 的 meta 中增加 `patch_96_mask` 字段，返回 96³ 的二值 mask tensor。

**影响文件**: `embed_to_full_ct.py`, `prompt_runner.py`

---

### 2.10 文件名 `_t`→`_s` 替换破坏 "tumor"

**问题**: `endometrioma_tumor_t00` 经 `_t`→`_s` 替换变为 `endometrioma_sumor_s00`。

**解决**: 使用正则 `_t(\d)→_s\1` 仅替换后接数字的 `_t`。

**影响文件**: `prompt_runner.py`, `main.py`, `batch_full_ct.py`

---

### 2.11 硬编码路径 → config.py 统一管理

**问题**: 所有源文件硬编码 `C:\Users\33067\...`，换机器必崩。

**解决**: 创建 `config.py`（自动检测项目根 + 加载 `paths.json`）+ `paths.json`（仅 4 行外部路径）。核心管线文件全部迁移至 config 导入。换机器只改 `paths.json`。

**影响文件**: `config.py`, `paths.json`, `prompt_runner.py`, `embed_to_full_ct.py`, `diffusion_engine.py`, `train_colon.py`

---

### 2.12 `repeat` 展开破坏随机 mask 选择

**问题**: task 展开逻辑 `t["mask_index"] = task.get("mask_index", 0) + j` 总是写入 `mask_index`，导致 `run_one_task` 中的随机 mask 选择逻辑被跳过——`repeat: 5` 每次都选同一个 mask。

**解决**: 仅当用户显式设了 `mask_index` 时才写入展开 task，否则保留为未设置状态让随机逻辑生效。

**影响文件**: `prompt_runner.py`

---

### 2.13 noearly (DDIM) 默认确定性 → 相同输入产出相同输出

**问题**: DDIM 默认 `eta=0` 是完全确定性的 ODE 求解器。相同条件 + 相同 mask → 完全相同输出。用户 `repeat` 多次看到相同图像。

**解决**: 将 `eta` 暴露为 `prompts.json` 字段（默认 0=论文标准，1=完全随机）。用户可按需选择。

**影响文件**: `diffusion_engine.py`, `embed_to_full_ct.py`, `prompt_runner.py`

---

### 2.14 `size_category` 仅作标签 → 改为物理尺寸过滤

**问题**: `size_category` 不控制实际 mask 选择——`large` 也可能选到 tiny mask。

**解决**: 扫描所有 mask 建立 `mask_size_index.json`，按体素体积反推半径，在 `resolve_mask` 中按档过滤。无匹配时自动降级（large→medium→small→tiny）。

**影响文件**: `prompt_runner.py`, `run_mask_gen.py`

---

### 2.15 输出文件覆盖 → 自动版本化

**问题**: 同一 mask 跑多次，文件名相同，直接覆盖。

**解决**: `save_full_ct` 和 patch 保存前检查文件是否存在，存在则追加 `_v2`、`_v3` 后缀。

**影响文件**: `embed_to_full_ct.py`, `prompt_runner.py`

---

### 2.16 Colon 训练内存泄漏 → 循环外复用

**问题**: `CTPreprocessor` 在每步训练循环内创建，SimpleITK C++ 对象累积 → 4.9 GB 内存 → 进程卡死。

**解决**: CTPreprocessor 移到循环外一次创建，每 100 步 `gc.collect()`，每步结束显式 `del` 大变量。

**影响文件**: `train_colon.py`

## 三、局限性

### 3.1 VQGAN 压缩伪影
VQGAN 以 4×4×4 倍压缩 CT，16384 个码本向量覆盖所有腹部纹理。在均质组织区域可能产生 subtle 网格/分块伪影。这是模型固有特性，非代码 bug。论文对此未特别讨论——分割训练下游模型对此鲁棒。

### 3.2 回采样损失
96³ 1mm³ 各向同性 patch 嵌入回原生 CT 时，线性/最近邻重采样引入约 40% 的对比度损失（80%→49% >20HU 体素）。这是物理必然——细粒度纹理被粗间距体素平均化。DiffTumor 的标准输出是 96³ patch（非 full-CT），分割训练直接使用 patch 格式可避免此损失。

### 3.3 零样本跨器官限制
esophagus/uterus 使用 liver_early.pt 零样本推理，仅支持早期小肿瘤。中晚期肿瘤（noearly）跨器官泛化未经论文验证。colon 已通过本地训练解决；esophagus/uterus 缺乏公开训练数据。

### 3.4 肿瘤对比度受模型阶段影响
- early 模型 (T=4 DDPM): 较明显，~63 HU 差异（纯 patch）
- noearly 模型 (T=200 DDIM): 较微弱，~28 HU 差异（纯 patch）
每次运行的随机噪声产生不同纹理（可暗可亮）。这是生成模型的固有随机性。

### 3.5 仅支持 6 种腹部器官
liver, pancreas, kidney, colon, esophagus, uterus。超出此范围需训练新权重。

### 3.6 无自然语言接口
当前输入为结构化 JSON 或 CLI 参数，不支持自然语言描述。需人工将"肝右叶 12mm 肿瘤"翻译为 JSON 字段。
